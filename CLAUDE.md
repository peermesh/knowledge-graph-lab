## 📋 UNIVERSAL STANDARDS

### Communication Style
- **Direct and Clear**: Skip pleasantries, get straight to solutions
- **Actionable**: Provide specific steps and examples users can follow
- **Context-Aware**: Ask clarifying questions when requirements are unclear
- **Professional**: Maintain helpful, knowledgeable tone without being verbose
- **Error-Handling**: Explain what went wrong and how to fix it
- **Teaching Mode**: Explain reasoning behind recommendations

### Handoff Task System

Quick reference:

- Priority: 1=🔴urgent, 2=🟡important, 3=🟢normal, 4=🔵low, 5=⚪nice-to-have

### Code Quality Standards
- **Readability**: Write self-documenting code with meaningful names
- **Error Handling**: Always implement proper error handling and user feedback
- **Testing**: Include unit tests and integration tests for all functionality
- **Documentation**: Comprehensive docstrings/comments for all public functions
- **Security**: Validate inputs, sanitize outputs, use secure defaults
- **Performance**: Profile before optimizing, focus on algorithmic improvements
- **Consistency**: Follow language-specific conventions and team standards

### Security Protocols
- **Input Validation**: Validate and sanitize all external inputs
- **Authentication**: Implement proper session management and access controls
- **Data Protection**: Encrypt sensitive data, use environment variables for secrets
- **Dependencies**: Keep dependencies updated, audit for vulnerabilities
- **Error Messages**: Don't expose sensitive information in error messages
- **Logging**: Log security events, but never log sensitive data

### Git Commit Standards (CRITICAL)
- **NO CO-AUTHOR ATTRIBUTION**: NEVER add "Co-Authored-By: Claude" or any AI attribution to commits
- **NO AI MARKERS IN COMMITS**: Do not add "🤖 Generated with Claude Code" or similar markers
- **USER'S WORK ONLY**: All commits belong solely to the user - AI assistants are tools, not contributors
- **CLEAN COMMIT MESSAGES**: Keep commit messages professional and focused on the changes made
- **OWNERSHIP**: The user owns 100% of their code and commits - AI must never claim credit

### AI Document Organization (MANDATORY)
- **ALL AI-Generated Files**: MUST be placed in `docs/ai/` subdirectory
  - **Definition**: ANY file created, written, or generated by an AI agent
  - **Includes**: Analysis reports, troubleshooting guides, implementation documentation, system documentation, architecture explanations, user manuals, API documentation, README files, configuration guides, etc.
  - **No Content-Based Exceptions**: Regardless of permanence, importance, intended audience, or perceived value to users
  - **ONLY Exception**: Files explicitly requested by the user to be placed elsewhere ("Please create README.md in the root directory")
- **File Extensions**: Use `.md` for all AI reference documents
- **Naming Convention**: `YY-MM-DD-HH-MM-[Tool]-[Description].md` format (e.g., `25-01-19-14-30-Claude-Code-analysis-report.md`)
  - `YY-MM-DD-HH-MM`: Timestamp (25-01-19-14-30 for Jan 19, 2025 at 14:30)
  - `[Tool]`: AI tool used (Claude-Code, Cursor, Windsurf, etc.)
  - `[Description]`: Brief kebab-case description
- **File Headers**: Include creation date, AI tool used, and purpose
- **NO EXCEPTIONS**: Never create AI documents in project root or random locations
- **Auto-Create Directory**: Create `docs/ai/` if it doesn't exist
- **Human-Readable**: All AI documents must be in markdown format for easy human review

#### Common Misclassifications to Avoid:
- ❌ "This is system documentation" → Still AI-generated, goes to docs/ai/
- ❌ "Users will reference this frequently" → Still AI-generated, goes to docs/ai/
- ❌ "This explains permanent architecture" → Still AI-generated, goes to docs/ai/
- ❌ "This matches existing docs/ files" → Those may predate the rule, still goes to docs/ai/
- ❌ "This is important project documentation" → Still AI-generated, goes to docs/ai/
- ❌ "This should be discoverable by developers" → Still AI-generated, goes to docs/ai/
- ✅ "The user explicitly asked me to create docs/README.md" → Exception allowed

#### Human vs AI Generated Rule:
- **If a human creates documentation**: Can be placed anywhere in the project
- **If an AI creates documentation**: MUST go to docs/ai/ with timestamp prefix
- **Mixed authorship**: If AI significantly contributes to content, treat as AI-generated
